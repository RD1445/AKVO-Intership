## đź”Ś 1. Supabase Client Integration

The database layer utilizes the official `supabase-py` SDK. The client connection is initialized in a thread-safe, centralized module:

* **File Location**: `app/database/supabase_client.py`
* **Mechanism**: On the first call, it retrieves authenticated environment variables (`SUPABASE_URL` and `SUPABASE_KEY`) and caches the active client connection instance. Subsequent queries call `get_supabase_client()` to reuse the singleton connection, preventing socket exhaustion.

The database is built around a centralized PostgreSQL table named **`esp_sensor_data`** that records real-time physical metrics, power metrics, operational states, and fault codes transmitted by AWG devices (or simulated by our test scripts).

## đź”Ť 3. Decoupling Database Queries (`app/database/queries.py`)

To ensure clean system architecture, raw database queries and filters are **never** executed inside API routers or scheduled cron jobs. Instead, database scripts are centralized in `queries.py`:

```python
# app/database/queries.py
from datetime import datetime, timedelta
from app.database.supabase_client import get_supabase_client

def fetch_telemetry_history(machine_id: str, hours: int = 24) -> list[dict]:
    """Retrieve all log entries for a given machine within a time window."""
    client = get_supabase_client()
    since = datetime.utcnow() - timedelta(hours=hours)
    
    response = client.table("esp_sensor_data") \
        .select("*") \
        .eq("machine_id", machine_id) \
        .gte("esp_log_at", since.isoformat()) \
        .order("esp_log_at", desc=True) \
        .execute()
        
    return response.data or []
```
* **Benefit**: If database names, columns, or backend connectors (e.g., swapping Supabase SDK for SQLAlchemy or Asyncpg) change, only this single file needs modification.

## đź›ˇď¸Ź 4. Business Services Layer (`app/services/`)

The business logic of the automation engine resides within the services layer. Rather than dumping complex analytical code inside FastAPI route handlers, functions are structured as pure service tasks.

### AWG Machine Health Analysis (`app/services/automation_health.py`)
This service parses recent historical logs for a machine to assess operational wellness. It checks for:
1. **Connectivity Loss**: Detects if a machine has failed to transmit log entries within a specific threshold (e.g., last 5 minutes).
2. **Component Integrity (Fan/Compressor Mismatch)**: Verifies that the evaporator fan is active whenever the compressor is running to prevent frozen coil blocks.
3. **Improper System Cycles**: Analyzes timestamps to detect if the compressor was started without the required fan preheat lead duration or stopped without fan cooldowns.
4. **Rapid Compressor Cycling**: Counts compressor toggles over short windows. Rapid cycling (e.g. switching on and off repeatedly) is a major hardware hazard and triggers immediate system alerts.
